1
SCAN — Découverte des fichiers .har
Le pipeline scanne data_dir et retourne la liste triée de tous les .har. Chaque fichier = un run = une exécution complète du scénario capturé par Playwright, k6 Browser ou équivalent.
2
EXTRACT — HARExtractor construit T1 / T2 / T3
HARExtractor(har_path, scenario_id) lit le JSON HAR et construit les 3 tableaux. Les clés FK (baseline_hash, step_baseline_key, scenario_id) sont générées ici directement — aucune étape add_keys séparée.
3
BUFFER — Insertion dans baseline_run_buffer
Chaque valeur métrique est insérée avec run_id, level et le flag is_incident. Le trigger de recalcul utilise COUNT(DISTINCT run_id) dans ce buffer — plus fiable que run_count_total.
4
RECALCUL — Mise à jour statistiques baseline
À chaque fenêtre SHORT (5 runs) ou LONG (100 runs) complète, les statistiques sont recalculées depuis le buffer avec winsorisation IQR (T1/T2) et stockées dans baseline_statistics.
5
ENRICH — Z-scores, robust_z, moyennes glissantes
Chaque tableau reçoit les features dérivées. Cold start (<5 obs) : donneur similaire. Partial (5–99 obs) : stats propres partielles. Complet (≥100) : baseline propre stable.
6
EXPORT — CSV / JSON + observabilité OBS_KEY
T1/T2/T3 enrichis exportés. Si OBS_KEY est défini, des dossiers horodatés capturent chaque étape (extract brut → baseline status → enrichi → résumé anomalie → pipeline global).

Extraction depuis le HAR

Section 07

Buffer Temporel

Le buffer baseline_run_buffer est la mémoire glissante du pipeline. Il stocke chaque valeur métrique brute avec son contexte d'exécution pour alimenter le recalcul des statistiques.

TABLE baseline_run_buffer
id PKBIGSERIAL — auto-incrémenté
baseline_keyVARCHAR(500) — clé entité (scénario/step/requête)
levelSCENARIO | STEP | REQUEST
metric_typenom de la métrique (ex: scenario_ttfb_ms_mean)
run_idVARCHAR(200) — identifiant unique du run (ISO timestamp)
run_timestampTIMESTAMPTZ — horodatage d'insertion
valueDOUBLE PRECISION — valeur numérique de la métrique
is_incidentBOOLEAN — TRUE si run labellisé comme incident

Règle sain / incident — ségrégation par niveau

Règle fondamentale : T1 (REQUEST) et T2 (STEP) n'insèrent dans leur baseline que les runs sains (is_incident=FALSE). T3 (SCENARIO) intègre tous les runs — la baseline scénario doit capturer le comportement complet, incidents inclus, pour XGBoost.
T1 REQUEST healthy_only=True
Seuls les runs sains alimentent la baseline. Un incident ne pollue pas la latence de référence des requêtes — la winsorisation renforce cette protection.
T2 STEP healthy_only=True
Même règle pour les steps. La baseline de référence d'un step reste celle des bonnes exécutions uniquement.
T3 SCENARIO tous runs
La baseline scénario intègre sains et incidents — supervisée pour XGBoost, elle doit refléter la réalité opérationnelle complète.

Trigger de recalcul — basé sur COUNT DISTINCT réel

# Compte les runs UNIQUES réellement insérés dans le buffer
# plus fiable que run_count_total (peut compter des runs échoués avant insertion)
SELECT COUNT(DISTINCT run_id) AS buf_count
FROM baseline_run_buffer
WHERE baseline_key = %s AND level = 'SCENARIO'

do_short = (buf_count - last_short_update_run) >= 5    # WINDOW_SHORT
do_long  = (buf_count - last_long_update_run)  >= 100   # WINDOW_LONG

Winsorisation IQR — protection outliers avant calcul (T1/T2 uniquement)

Winsorisation k=3.0 — préserve ~99.7% des valeurs d'une distribution normale lo = Q1 − 3 × IQR · hi = Q3 + 3 × IQR
valeur_winsorisée = clip(valeur, lo, hi)
Appliquée sur les valeurs du buffer avant _stats() → protège mean et std contre les pics incidents

Section 10

Fallback & Similarité

Quand une entité a moins de 5 observations, le pipeline cherche le donneur le plus similaire parmi ceux ayant ≥100 observations. Un donneur partiel ne peut jamais servir de référence.

Score de similarité — maximum 13 points

request_category (même type)
+5
Distributions quasi-identiques
is_api (API vs statique)
+3
Latences fondamentalement différentes
is_third_party (domaine tiers)
+2
Réseau externe non contrôlé
same_scenario (même scénario)
+2
Même contexte navigateur
step_proximity (position step)
+1
Position dans le waterfall
Condition stricte : Un donneur n'est éligible que s'il a lui-même atteint MIN_OBS = 100 observations. Une baseline partielle (5–99 obs) ou un autre cold start ne peut jamais être donneur — seules les baselines complètes et stables sont autorisées (s.count >= self.min_obs).

Graduation automatique — sortie du cold start

# Après chaque recalcul LONG : supprimer du cache fallback les entités
# qui ont atteint MIN_OBS dans baseline_statistics (elles utiliseront
# leur propre baseline dès le prochain run)
DELETE FROM baseline_fallback_cache fc
WHERE EXISTS (
    SELECT 1 FROM baseline_statistics s
    WHERE s.baseline_key = fc.baseline_key
      AND s.level        = fc.level
      AND s.count >= 100   -- MIN_OBS
)
Section 11

Z-Scores & Enrichissement

Chaque tableau est enrichi en comparant les valeurs du run courant à la baseline. Deux z-scores simultanés par métrique : classique (sensible aux pics) et robuste (résistant aux outliers).

Z-Score Classique
Formulez = (val − mean) / std
Basé sur la moyenne et l'écart-type. Sensible aux outliers — un seul incident peut gonfler mean/std et réduire la détectabilité future. Winsorisation du buffer atténue ce problème.
Robust Z-Score v4
Formule robusterobust_z = (val − median) / robust_std
Basé sur la médiane et IQR × 0.7413. Insensible aux outliers. Détecte les anomalies dans les distributions asymétriques — typique des latences réseau avec queues longues.
Score d'anomalie agrégé — par niveau z_eff(m) = max( |z_score(m)|, |robust_z(m)| ) ← prend le pire des 2
anomaly_score = moyenne des z_eff sur les métriques clés du niveau

Seuils d'alerte is_anomaly_warn = 1 si anomaly_score ≥ 2.0 (Z_WARN — déviation notable)
is_anomaly_alert = 1 si anomaly_score ≥ 3.0 (Z_ALERT — anomalie franche)

Gestion de la Criticité

Chaque entité possède une criticité dynamique reflétant son historique d'incidents. Elle augmente sur chaque incident (+0.1) et décroît progressivement vers son défaut (−0.01 par période LONG sans incident).

Bump — lors de chaque run incident

# Appelé dans update() pour CHAQUE run labellisé is_incident=True
# 1. Bump le SCÉNARIO (plafond = 1.0)
UPDATE baseline_metadata
SET criticality_current = LEAST(1.0, criticality_current + 0.1),
    incident_occurrences = incident_occurrences + 1,
    last_updated = NOW()
WHERE baseline_key = scenario_id AND level = 'SCENARIO'

# 2. Bump chaque STEP impliqué (loop sur t2)
FOR step IN t2:
    UPDATE baseline_metadata
    SET criticality_current = LEAST(1.0, criticality_current + 0.1),
        incident_occurrences = incident_occurrences + 1
    WHERE baseline_key = step_hashed_key(scenario_id, step.index) AND level = 'STEP'

# 3. Bump chaque REQUEST unique (dédupliquées par baseline_hash)
FOR req IN deduplicated(t1, key='baseline_hash'):
    UPDATE baseline_metadata
    SET criticality_current = LEAST(1.0, criticality_current + 0.1),
        incident_occurrences = incident_occurrences + 1
    WHERE baseline_key = req.baseline_hash AND level = 'REQUEST'

Decay — à chaque recalcul LONG sans incident

# Déclenché dans update() à chaque recalcul LONG (100 runs) si run non-incident
# Toutes les entités du scénario redescendent vers leur défaut
UPDATE baseline_metadata
SET criticality_current = GREATEST(
    criticality_default,           ← plancher = jamais en dessous du défaut initial
    criticality_current - 0.01    ← -1% par période LONG (= 100 runs)
),
last_updated = NOW()
WHERE scenario_id = %s
  AND criticality_current > criticality_default
