Architecture Finale · Détection d'Incidents

Système Hybride de Détection
d'Incidents à partir de Fichiers HAR

Un pipeline en deux étapes : Modèle 1 score chaque requête HTTP individuellement par rapport à sa baseline contextuelle, puis Modèle 2 agrège ces scores avec les features du scénario pour décider s'il y a un incident.

Vue d'ensemble

Pipeline Complet

Depuis un fichier HAR unique, trois tableaux sont construits simultanément. Modèle 1 s'entraîne sur T1, scores agrégés → T3 enrichi → Modèle 2 décide.

Un fichier HAR = un scénario complet de navigation (connexion → consultation → action). Contient toutes les requêtes HTTP horodatées avec leurs timings détaillés.

Étape clé

🔑 Génération de la Clé Baseline

Pour chaque ligne de T1, on calcule une clé composite qui identifie le type fonctionnel de la requête : méthode + domaine + path normalisé + paramètres fonctionnels. Cette clé permet de comparer chaque requête uniquement à ses semblables, pas à toutes les requêtes du HAR.

baseline_key = hash( method + domain + path_normalized + functional_params_keys + resource_type )

Pourquoi trois tableaux ? T1 permet le scoring fin par requête (contexte individuel). T2 capture la dégradation progressive entre étapes (pattern temporel). T3 fournit la vue globale pour la décision finale. Chaque tableau a un rôle distinct dans la chaîne de décision.
Problème central

La Clé Baseline — Comparer le Comparable

Le problème : Un CSS répond normalement en 20 ms, une API de paiement en 800 ms. Sans segmentation, le modèle traite tout ensemble et rate les vraies anomalies. La solution : comparer chaque requête uniquement à ses semblables historiques.

Algorithme de construction de la clé

import re, hashlib
from urllib.parse import urlparse

# Paramètres à filtrer : techniques, anti-cache, session
EXCLUDE_PARAMS = {
    'tabId', '_t', 'timestamp', '_cache', 'rand', 'nonce',
    'sessionId', 'token', 'JSESSIONID', 'v', 'version', '_',
    'cb', 'callback', 'requestId', 'correlationId'
}

# Patterns d'IDs variables dans les paths
ID_PATTERNS = [
    (re.compile(r'/\d{5,}'),      '/<ID>'),
    (re.compile(r'/[a-f0-9]{8,}'), '/<HASH>'),
    (re.compile(r'/[A-Z]{2}\d{6,}'),'/<REF>'),
    (re.compile(r'/v\d+/'),         '/vN/'),
]

def normalize_path(path: str) -> str:
    for pattern, replacement in ID_PATTERNS:
        path = pattern.sub(replacement, path)
    return path.lower().rstrip('/')

def filter_functional_params(qs_list: list) -> list:
    return sorted(
        p['name'] for p in qs_list
        if p['name'].lower() not in EXCLUDE_PARAMS
    )

def build_baseline_key(entry: dict) -> str:
    req     = entry.get('request', {})
    url     = req.get('url', '')
    parsed  = urlparse(url)
    method  = req.get('method', '').upper()
    domain  = parsed.netloc.lower()
    path    = normalize_path(parsed.path)
    qs_keys = filter_functional_params(req.get('queryString', []))
    pd_keys = filter_functional_params(
        (req.get('postData', {}) or {}).get('params', []))
    rtype   = infer_resource_type(entry)  # depuis har_extractor

    raw = f"{method}|{domain}|{path}|{','.join(qs_keys)}|{','.join(pd_keys)}|{rtype}"
    return hashlib.md5(raw.encode()).hexdigest()[:12]  # 12 chars lisibles

# Exemples de clés générées :
# GET|api.gen.fr|/api/v2/contract/<ID>||action|fetch  → "3f9a2c1e8d7b"
# GET|cdn.gen.fr|/static/main.min.css|||stylesheet     → "a14b56f2c8e3"
# POST|auth.gen.fr|/auth/login|username,password||doc  → "92d3f1a47e5c"
Exemple concret :
• /api/v2/contracts/12345678 et /api/v2/contracts/98765432 → même clé /api/v2/contracts/<ID>
• /user/auth/dologin?tabId=123456 → tabId filtré → même clé que /user/auth/dologin?tabId=999999
• GET /main.css ≠ GET /api/data → clés différentes → baselines séparées
Système de référence

Double Baseline par Clé

Baseline Figée

📌 50 derniers runs normaux

Référence stable constituée sur 50 exécutions sans incident. Absorbe la dérive naturelle sur ~3 mois. Réinitialisée manuellement après une MEP majeure.

baseline_fixed[key] = {
mean_ttfb: np.mean(history[-50:]['timing_wait_ms']),
std_ttfb: np.std(history[-50:]['timing_wait_ms']),
p95_total: np.percentile(history[-50:]['timing_total_ms'], 95),
mean_body: np.mean(history[-50:]['response_body_size']),
n_samples: len(history[-50:]),
if_model: IsolationForest().fit(history[-50:])
}
Baseline Glissante

🔄 10 derniers runs normaux

Fenêtre courte qui suit l'évolution récente de l'application. Détecte les dérives progressives imperceptibles par la baseline figée.

baseline_sliding[key] = {
mean_ttfb: np.mean(history[-10:]['timing_wait_ms']),
std_ttfb: np.std(history[-10:]['timing_wait_ms']),
p95_total: np.percentile(history[-10:]['timing_total_ms'], 95),
n_samples: len(history[-10:]),
if_model: IsolationForest().fit(history[-10:])
}
Stratégie d'activation : On utilise les deux scores en parallèle comme features pour le Modèle 2. Si score_fixed élevé → anomalie absolue. Si score_sliding élevé mais score_fixed normal → dérive progressive récente. Les deux ensemble = signal plus riche.

Modèle 1 · Per-Request

Principe : Un Isolation Forest distinct est entraîné pour chaque clé baseline ayant ≥ 30 échantillons. Si la clé a moins de 30 échantillons, on utilise un modèle global de fallback entraîné sur toutes les requêtes du même resource_type.
